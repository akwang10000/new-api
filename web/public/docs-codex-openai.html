如何把 Codex 接到 OpenAI 兼容接口

Codex CLI 可以直接通过自定义 provider 配置连接 OpenAI 兼容接口，不需要额外翻译层。对 routeropenai 来说，关键是把 provider 名称、令牌变量和 `https://routeropenai.xyz/v1` 配对正确。

真正的重点

要对齐 3 个位置

最关键的是 provider 名称、`base_url` 和对应环境变量；这三处只要对齐，链路就很清晰。

推荐做法

单独建一个 provider

不要覆盖默认配置，给 routeropenai 单独留一个 provider，后面切模型或切服务更稳。

前置条件

  1. 你已经拿到一个可用的 routeropenai 令牌。
  2. 站点通过 `https://routeropenai.xyz/v1` 提供 OpenAI 兼容接口。
  3. 本机已经安装 Codex CLI。
说明： 本页使用的就是本站真实可用的接口地址，不是占位 URL。

步骤 1：配置 Codex provider

在 `~/.codex/config.toml` 里新增或修改下面这段配置：

model_provider = "routeropenai"
model = "gpt-5.4"
model_reasoning_effort = "high"
personality = "pragmatic"

[model_providers.routeropenai]
name = "routeropenai"
base_url = "https://routeropenai.xyz/v1"
env_key = "ROUTEROPENAI_API_KEY"
  • `model_provider = "routeropenai"` 表示让 Codex 使用自定义 provider。
  • `base_url` 必须指向 `/v1` API 根路径，而不是首页域名。

步骤 2：注入访问令牌

当前示例使用 `env_key = "ROUTEROPENAI_API_KEY"`，所以启动 Codex 的终端里要提供同名环境变量：

# macOS / Linux
export ROUTEROPENAI_API_KEY=sk-your-routeropenai-token

# Windows PowerShell
$env:ROUTEROPENAI_API_KEY="sk-your-routeropenai-token"
不要混淆： 这里不要求必须叫 `OPENAI_API_KEY`，真正读取哪个变量名，取决于 `env_key`。

步骤 3：启动 Codex

配置完成后，直接在目标仓库里启动 Codex 即可：

codex

# 或非交互模式
codex exec "检查当前项目的 README 并给出接入建议"

如果 Codex 能正常进入会话，而不是立刻报认证错误或 provider 配置错误，说明基础链路已经打通。

验证点

检查 1

Codex 能正常启动

启动后应能进入交互界面，或者正常执行 `codex exec`。

检查 2

routeropenai 有请求记录

后台应该能看到对应模型调用、日志或计费记录。

检查 3

模型名称有效

如果 provider 已通但模型名无效，请求还是会失败，这时要改回站点已开放的模型。

常见问题

  • 把 `base_url` 写成首页地址，而不是带 `/v1` 的接口根路径。
  • `env_key` 和实际环境变量名不一致，导致 Codex 读不到令牌。
  • 把 Claude Code 那套 gateway 方案照搬给 Codex，多加了一层不必要的中转。
  • 模型名称填成站点未开放或未映射的模型，表现为连通了但请求失败。

推荐做法

  1. 给 routeropenai 单独保留一个 `routeropenai` provider。
  2. 把 `base_url` 固定写成 `https://routeropenai.xyz/v1`。
  3. 用独立环境变量保存令牌，例如 `ROUTEROPENAI_API_KEY`。
  4. 先用一个稳定模型验证联通，再切到你真正要用的模型。
一句话总结： Codex 接入 routeropenai 的关键不是协议转换，而是把 provider、`/v1` 地址和令牌变量配正确。
